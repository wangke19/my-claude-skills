# Retry Strategy Presets

本文档提供针对不同测试时长的推荐重试策略配置。

## 核心原则

对于长时间运行的 CI 测试任务，设置重试策略需要平衡：

1. **成本控制**：每次重试都是高成本（时间 + 资源）
2. **成功率提升**：合理重试可以显著提高最终成功率
3. **总时间边界**：必须设定明确的超时上限

## 策略设计思路

```
用尽可能少的高成本重试次数，换取尽可能高的成功率，同时必须设定明确的总超时边界。
```

关键考虑因素：
- **单次运行时长**：决定检查间隔和回退基数
- **失败时间分散性**：决定是否需要长回退等待基础设施恢复
- **资源竞争风险**：决定是否需要随机抖动避免"惊群效应"

## 预设配置

### 1. 短时测试 (< 30分钟)

**适用场景**：
- 单元测试、集成测试
- 快速 E2E 测试
- 失败通常是偶发的网络问题或资源瞬时不可用

**推荐配置**：
```bash
export CI_MONITOR_CHECK_INTERVAL=300      # 5分钟
export CI_MONITOR_MAX_RETRIES=3           # 3次重试
export CI_MONITOR_BASE_BACKOFF=300        # 5分钟基础回退
export CI_MONITOR_BACKOFF_JITTER=10       # ±10% 抖动
```

**时间线**：
```
T+0min:   首次运行开始
T+20min:  首次失败
T+25min:  重试1开始 (等待5分钟 ± 30秒)
T+45min:  重试1失败
T+55min:  重试2开始 (等待10分钟 ± 1分钟)
T+75min:  重试2失败
T+95min:  重试3开始 (等待20分钟 ± 2分钟)
T+115min: 结束
总计：    约2小时，4次运行
```

**指数回退序列**：
```
重试1: 5分钟  (5 × 2^0)
重试2: 10分钟 (5 × 2^1)
重试3: 20分钟 (5 × 2^2)
```

---

### 2. 中等时长测试 (30分钟 - 2小时)

**适用场景**：
- 中型 E2E 测试
- 性能测试
- 偶发的基础设施问题

**推荐配置**：
```bash
export CI_MONITOR_CHECK_INTERVAL=600      # 10分钟
export CI_MONITOR_MAX_RETRIES=2           # 2次重试
export CI_MONITOR_BASE_BACKOFF=1200       # 20分钟基础回退
export CI_MONITOR_BACKOFF_JITTER=10       # ±10% 抖动
```

**时间线**：
```
T+0h:    首次运行开始
T+1h:    首次失败
T+1.3h:  重试1开始 (等待20分钟 ± 2分钟)
T+2.3h:  重试1失败
T+3h:    重试2开始 (等待40分钟 ± 4分钟)
T+4h:    结束
总计：   约4小时，3次运行
```

**指数回退序列**：
```
重试1: 20分钟 (20 × 2^0)
重试2: 40分钟 (20 × 2^1)
```

---

### 3. 长时测试 (2-4小时) ⭐ **默认配置**

**适用场景**：
- 大型 E2E 测试套件
- 升级测试
- 失败时间点分散（可能在任何阶段失败）
- 基础设施需要较长恢复时间

**推荐配置**：
```bash
export CI_MONITOR_CHECK_INTERVAL=1800     # 30分钟
export CI_MONITOR_MAX_RETRIES=2           # 2次重试
export CI_MONITOR_BASE_BACKOFF=3600       # 60分钟基础回退
export CI_MONITOR_BACKOFF_JITTER=10       # ±10% 抖动
```

**时间线**：
```
T+0h:    首次运行开始
T+4h:    首次失败
T+5h:    重试1开始 (等待60分钟 ± 6分钟)
T+9h:    重试1失败
T+11h:   重试2开始 (等待120分钟 ± 12分钟)
T+15h:   结束
总计：   约15小时，3次运行
```

**指数回退序列**：
```
重试1: 60分钟  (60 × 2^0)
重试2: 120分钟 (60 × 2^1)
```

**为什么是这个配置？**
1. **2次重试**：平衡成本与成功率
   - 1次重试：成功率提升 ~70%
   - 2次重试：成功率提升 ~90%
   - 3次重试：边际收益递减，成本过高

2. **60分钟基础回退**：
   - 给基础设施充足恢复时间
   - 避免在问题未解决时立即重试
   - 120分钟二次回退对严重问题足够

3. **30分钟检查间隔**：
   - 对4小时任务足够（检查8次）
   - 不会产生过多无意义的状态查询

4. **±10% 抖动**：
   - 避免多个失败任务同时重试
   - 减轻资源竞争

---

### 4. 超长时测试 (> 4小时)

**适用场景**：
- 升级/降级完整流程
- 大规模压力测试
- 多集群互操作测试
- 高成本、不容轻易重试

**推荐配置**：
```bash
export CI_MONITOR_CHECK_INTERVAL=3600     # 60分钟
export CI_MONITOR_MAX_RETRIES=1           # 仅1次重试
export CI_MONITOR_BASE_BACKOFF=7200       # 120分钟基础回退
export CI_MONITOR_BACKOFF_JITTER=15       # ±15% 抖动
```

**时间线**：
```
T+0h:    首次运行开始
T+6h:    首次失败
T+8h:    重试1开始 (等待120分钟 ± 18分钟)
T+14h:   结束
总计：   约14小时，2次运行
```

**指数回退序列**：
```
重试1: 120分钟 (120 × 2^0)
```

**策略说明**：
- **极度保守**：仅1次重试，避免过高成本
- **长回退**：2小时等待确保基础设施完全恢复
- **大抖动**：±15% 避免批量任务冲突

## 高阶优化：基于失败类型的动态策略

### 早期失败 (< 30分钟内)

如果任务在运行初期失败，通常是：
- 环境配置问题
- 连接失败
- 资源分配失败

**建议**：缩短回退时间
```bash
# 对于早期失败，可以更快重试
BASE_BACKOFF=1800  # 30分钟而不是60分钟
```

### 中后期失败 (> 2小时后)

如果任务运行很久后失败，通常是：
- 资源耗尽（内存、磁盘）
- 累积的状态问题
- 罕见的竞态条件

**建议**：延长回退时间
```bash
# 对于后期失败，需要更长恢复时间
BASE_BACKOFF=7200  # 120分钟确保环境完全重置
```

### 实现示例

在 `classify_failure.sh` 中添加：

```bash
# 提取失败时间点
failure_time=$(extract_failure_timestamp)
runtime=$((failure_time - start_time))

if [ $runtime -lt 1800 ]; then
    # 早期失败 (< 30分钟)
    echo "EARLY_FAILURE"
elif [ $runtime -gt 7200 ]; then
    # 后期失败 (> 2小时)
    echo "LATE_FAILURE"
else
    echo "MID_FAILURE"
fi
```

## 随机抖动 (Jitter) 详解

### 为什么需要抖动？

**问题场景**：
```
假设10个相同的CI任务同时开始运行：
T+0h:   10个任务同时开始
T+4h:   10个任务同时失败（同样的基础设施问题）
T+5h:   10个任务同时重试 ← 惊群效应！资源再次耗尽！
```

**有了抖动**：
```
T+4h:   10个任务同时失败
T+5h:   任务1重试 (等待54分钟)
T+5.1h: 任务2重试 (等待58分钟)
T+5.2h: 任务3重试 (等待63分钟)
...     分散的重试时间，避免资源竞争
```

### 抖动大小建议

| 测试时长 | 推荐抖动 | 原因 |
|---------|---------|------|
| < 30分钟 | ±5-10% | 小抖动足够，避免过度延迟 |
| 30分钟-2小时 | ±10% | 标准配置 |
| 2-4小时 | ±10-15% | 较大抖动避免批量任务冲突 |
| > 4小时 | ±15-20% | 最大抖动，因单次成本极高 |

### 抖动实现

当前脚本实现：
```bash
# ±10% 抖动示例
backoff_base=3600  # 60分钟
jitter_range=360   # ±6分钟 (10%)
jitter=$(random -360 to +360)
actual_backoff=3600 + jitter  # 54-66分钟之间随机
```

## 选择配置的决策树

```
测试运行时长？
│
├─ < 30分钟    → 使用"短时测试"配置 (3次重试, 5分钟回退)
│
├─ 30分钟-2小时 → 使用"中等时长"配置 (2次重试, 20分钟回退)
│
├─ 2-4小时     → 使用"长时测试"配置 ⭐ (2次重试, 60分钟回退)
│
└─ > 4小时     → 使用"超长时测试"配置 (1次重试, 120分钟回退)
```

额外考虑：
- **资源成本高？** → 减少重试次数
- **失败率很高？** → 增加重试次数，但评估是否有系统性问题
- **多任务并发？** → 增加抖动百分比
- **关键路径？** → 可以适当增加1次重试

## 成功率估算

假设单次基础设施失败概率为 30%：

| 重试次数 | 预期成功率 | 总运行时间（4h任务） | 成本 |
|---------|-----------|-------------------|------|
| 0（不重试） | 70% | 4h | 1× |
| 1次重试 | 91% | 9h | 2.25× |
| 2次重试 | 97.3% | 15h | 3.75× |
| 3次重试 | 99.2% | 22h | 5.5× |

**推荐**：2次重试是成本与收益的最佳平衡点。

## 总结

- **默认配置已优化为长时测试**（2-4小时任务）
- 根据实际测试时长选择预设配置
- 使用环境变量灵活调整
- 监控实际重试成功率，持续优化配置
- 当成功率 > 95% 时，考虑减少重试；< 85% 时，考虑增加或排查系统性问题
